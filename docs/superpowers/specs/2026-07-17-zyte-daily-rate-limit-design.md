# Zyte 연동 API 일일 호출 제한 설계

## 배경

2026년 7월 Zyte 월간 사용액이 $100 한도의 85%에 도달했다. Cloudflare 요청 통계를 확인한 결과, 비용 증가는 큰 응답 본문이나 브라우저 렌더링이 아니라 소수 IP의 반복 호출과 캐시 우회에서 발생했다.

- 2026년 7월 10일부터 17일까지 `/api/cgv/timetable` GET 요청은 212,887회였다.
- 7월 16일 CGV 시간표 요청 70,608회 중 69,175회가 미국에서 발생했고, IP 6개가 거의 전부를 차지했다.
- 같은 날 가장 많이 호출한 IP 하나가 CGV 시간표를 54,959회 요청했다.
- `/api/oliveyoung/products`는 최근 8일 동안 70,231회 요청됐다.
- 실시간 로그에서 한 `node` 클라이언트가 서로 다른 브랜드명을 사용해 올리브영 상품 검색을 수초 간격으로 순회했다.
- 올리브영 요청은 처음부터 Zyte를 사용한다. CGV 원본 API는 현재 403을 반환하므로 캐시 미스마다 Zyte fallback을 사용한다.
- Zyte 공통 클라이언트는 재시도를 한 번 허용하므로 일부 실패는 최대 두 번의 Zyte 호출로 늘어난다.

## 목표

- 소수 클라이언트가 Zyte 연동 API를 대량 호출해 월간 프록시 비용을 소진하지 못하게 한다.
- 정상적인 익명 사용자는 별도 API 키 없이 계속 사용할 수 있게 한다.
- 캐시 적중 여부와 관계없이 클라이언트별 일일 호출량을 제한한다.
- 헬스 체크와 MCP 스트림·세션 통신은 제한 대상에서 제외한다.
- 원본 IP를 저장하지 않고 제한 상태만 유지한다.

## 제한 정책

### 적용 요청

다음 경로의 GET 요청을 한 IP의 공통 일일 할당량으로 합산한다.

- `/api/oliveyoung/*`
- `/api/cgv/*`
- `/api/cu/*`
- `/api/gs25/*`
- `/api/lottemart/*`

한 IP가 위 경로 전체에서 KST 기준 하루 3,000회까지 호출할 수 있다. 3,001번째 요청부터 다음 KST 자정까지 HTTP 429를 반환한다. 캐시 적중 요청, 유효성 검사 실패 요청, upstream 실패 요청도 모두 한 회로 계산한다.

### 제외 요청

다음 요청은 카운터를 조회하거나 증가시키지 않는다.

- `/health`
- `/api/health/checks`
- `/mcp`의 스트림과 세션 통신
- 루트 경로의 MCP POST·DELETE 통신
- OPTIONS와 GET 이외의 메서드
- 정적 문서, OpenAPI, 프롬프트, 개인정보 처리방침 경로
- 위 적용 목록에 없는 API

헬스 체크가 보호 대상 API를 내부 호출하는 경우에는 서버가 검증할 수 있는 내부 헬스 체크 표식을 전달해 제한을 우회한다. 외부 사용자가 임의로 표식을 만들 수 없도록 기존 Worker secret을 사용한다.

## 구현 구조

### Durable Object 분산 카운터

정확한 일일 한도를 위해 SQLite 저장소를 사용하는 Durable Object를 추가한다. Workers Rate Limiting binding은 10초 또는 60초 창만 지원하므로 일일 3,000회 정책을 정확하게 표현할 수 없다.

전역 객체 하나에 모든 IP를 모으지 않는다. 요청 IP의 SHA-256 해시로 Durable Object ID를 생성해 IP마다 별도 객체를 사용한다. 이 구조는 IP별 요청을 직렬화하면서 서로 다른 IP의 트래픽은 분산 처리한다.

객체 저장소에는 다음 값만 둔다.

- 현재 KST 날짜
- 해당 날짜의 호출 횟수

KST 날짜가 바뀐 첫 요청에서 횟수를 0으로 초기화한 뒤 새 요청을 계산한다. 날짜별 객체를 새로 만들지 않으므로 오래된 일별 객체가 계속 쌓이지 않는다.

### 요청 흐름

1. Worker가 요청 메서드와 경로를 확인한다.
2. 제외 요청이면 기존 라우팅을 그대로 진행한다.
3. 보호 대상 GET이면 `CF-Connecting-IP`를 읽는다.
4. IP를 SHA-256으로 해시하고 해당 Durable Object를 호출한다.
5. 카운터가 3,000 미만이면 한 회 증가시키고 기존 라우팅과 캐시 처리를 진행한다.
6. 이미 3,000회에 도달했으면 handler와 edge cache에 진입하지 않고 429를 반환한다.

`CF-Connecting-IP`가 없는 요청은 여러 사용자를 한 카운터로 잘못 묶지 않도록 제한 없이 허용한다. 로컬·통합 테스트에서는 요청 헤더에 고정된 테스트 IP를 넣어 제한 동작을 검증한다.

## 응답 형식

허용된 보호 요청에는 다음 헤더를 추가한다.

- `X-RateLimit-Limit: 3000`
- `X-RateLimit-Remaining: <남은 횟수>`
- `X-RateLimit-Reset: <다음 KST 자정의 Unix timestamp>`

초과 요청은 기존 API 오류 형식에 맞춘 JSON과 HTTP 429를 반환한다.

```json
{
  "success": false,
  "error": {
    "code": "DAILY_RATE_LIMIT_EXCEEDED",
    "message": "이 IP의 일일 호출 한도 3,000회를 초과했습니다."
  }
}
```

429 응답에는 위 세 헤더와 함께 `Retry-After`를 다음 KST 자정까지 남은 초 단위 값으로 설정한다.

## Zyte 관측성 보강

Zyte 공통 요청 옵션에 `tags`를 추가하고 호출 서비스와 기능을 태그로 전달한다. 최소한 다음 서비스 태그를 구분한다.

- `service:oliveyoung`
- `service:cgv`
- `service:cu`
- `service:gs25`
- `service:lottemart`

태그는 Zyte Stats API에서 비용과 성공 요청을 서비스별로 필터링하기 위한 용도이며 호출 동작이나 응답에는 영향을 주지 않는다. 기능 태그는 현재 호출 경계에서 명확히 구분되는 경우에만 추가하고, 불명확한 추론값은 기록하지 않는다.

## 오류 처리

- Durable Object가 일시적으로 실패하면 정상 사용자 전체가 차단되지 않도록 요청을 허용한다. 다만 제한 검사가 실패했다는 사실은 관측 가능한 오류로 남긴다.
- Durable Object가 허용한 뒤 upstream에서 실패해도 이미 사용한 한 회는 되돌리지 않는다.
- 응답에 rate limit 헤더를 추가할 수 없는 스트리밍 응답은 애초에 제한 대상에서 제외한다.
- 내부 헬스 체크 우회 표식이 유효하지 않으면 일반 요청과 똑같이 계산한다.

## 개인정보와 보안

- 원본 IP는 저장소, 애플리케이션 로그, 응답에 기록하지 않는다.
- SHA-256 해시값은 Durable Object 라우팅 ID를 만드는 용도로만 사용한다.
- rate limit 응답으로 다른 IP의 사용량을 조회할 수 없게 한다.
- 내부 헬스 체크 우회에는 새 공개 토큰을 만들지 않고 기존 secret을 재사용한다.

## 테스트

테스트는 구현보다 먼저 작성하며 다음 동작을 검증한다.

- 보호 대상 GET은 카운터를 한 회 증가시킨다.
- 다섯 서비스 경로가 동일한 IP의 공통 할당량을 사용한다.
- 캐시 적중 여부와 무관하게 카운터가 증가한다.
- 3,000번째 요청은 허용하고 3,001번째 요청은 429를 반환한다.
- KST 날짜가 바뀌면 카운터가 초기화된다.
- 서로 다른 IP는 독립된 카운터를 사용한다.
- 원본 IP가 저장 값과 응답에 포함되지 않는다.
- 헬스 체크, MCP 스트림·세션, OPTIONS, 보호 목록 밖의 API는 카운터를 호출하지 않는다.
- 유효한 내부 헬스 체크는 보호 대상 API를 호출해도 카운터를 사용하지 않는다.
- Durable Object 실패 시 요청을 허용하는 fail-open 정책이 동작한다.
- 허용 응답과 429 응답의 rate limit 헤더가 올바르다.
- Zyte 요청에 서비스 태그가 전달된다.

전체 `npm run check`, `npm run test:coverage`, 빌드와 로컬 Worker 통합 테스트를 통과해야 한다.

## 배포와 검증

- Durable Object migration과 binding을 `wrangler.toml`에 선언한다.
- 코드와 설정은 draft PR에서 검토한다.
- 배포 후 보호 대상 API가 정상적으로 응답하고 rate limit 헤더가 붙는지 확인한다.
- 테스트 전용 IP 카운터나 낮은 임계값으로 운영 데이터를 오염시키지 않는다.
- Cloudflare 요청 통계와 Zyte Stats를 하루 뒤 다시 확인해 CGV·올리브영 호출량과 비용 감소를 검증한다.

## 비목표

- API 키 기반 유료 요금제나 사용자 계정 시스템을 도입하지 않는다.
- 국가, 사용자 에이전트, 특정 IP를 영구 차단하지 않는다.
- MCP 스트림·세션 프로토콜의 메시지 수를 제한하지 않는다.
- Zyte 월간 지출 한도를 자동 변경하지 않는다.
- 현재 Zyte 재시도 정책은 이번 변경에서 조정하지 않는다.
